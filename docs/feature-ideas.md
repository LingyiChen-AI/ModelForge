# ModelForge 功能演进建议

> 基于当前代码现状(2026-06-14)梳理的可做功能清单。每条标注**价值**、**落地要点**(改哪些服务/模型/是否需要编号迁移)、**工作量**(S/M/L)与**优先级**。
>
> 现状基线:数据集(3 类 + 版本快照)、模型容器、训练(classification/ner/pair/embedding,实时进度 + MLflow)、模型测试(进度 + Leaderboard)、在线部署(启/停/重启 + API 详情 + 删除)、RBAC(自定义角色 + 数据范围)、Dashboard 统计、model-server 统一 `{code,data,message}` 信封。
>
> 约束提醒:任何动 `app/models/**` 的改动都必须**同包配一个编号 SQL 迁移**(见根 `CLAUDE.md`)。

---

## 一、推理服务层(对外能力,投入产出比最高)

当前 model-server 只有 JWT 内部调用,缺少"把模型当成产品 API 对外提供"的关键能力。

| # | 功能 | 价值 | 落地要点 | 工作量 | 优先级 |
|---|---|---|---|---|---|
| 1.1 | **API Key / 访问令牌** | 业务方无需登录账号即可程序化调用;可按 key 限权、吊销 | 新增 `ApiKey` 表(`+ 迁移`)、`X-API-Key` 鉴权依赖、model-server 校验;UI 管理页(创建/复制/吊销) | M | **P0** |
| 1.2 | **推理调用日志 + 用量统计** | 谁、什么时候、调了哪个部署多少次、延迟多少;计费/排障基础 | model-server 落 `InferenceLog`(或异步写 PG/Redis 计数)、Dashboard 增加调用量曲线、部署详情页看 QPS/P95 | M | **P0** |
| 1.3 | **限流 / 配额** | 防止单业务方打爆 worker;多团队公平 | 基于 API Key 的令牌桶(Redis)、超限返回信封 `code=429` | S | P1 |
| 1.4 | **批量 / 异步推理** | 大批量离线打分不占在线接口;走 Celery | `POST /batch-predict` 建任务 → worker 批跑 → 结果落对象存储下载 | M | P1 |
| 1.5 | **金丝雀 / A·B 多版本路由** | 新版本灰度,按比例分流,观察指标再全量 | 部署支持挂多个 `ModelVersion` + 权重;model-server 按权重选版本;`+ 迁移` | L | P2 |
| 1.6 | **多副本 / 自动伸缩** | 高并发场景横向扩展 | model-server 无状态化 + 进程池/多实例 + 负载均衡;部署记录副本数 | L | P2 |

## 二、评估深度(把"算个指标"升级为"看懂模型")

`EvalRun.per_sample_uri` 字段已存在但前端未消费——这是最低垂的果子。

| # | 功能 | 价值 | 落地要点 | 工作量 | 优先级 |
|---|---|---|---|---|---|
| 2.1 | **逐样本明细 + 错误分析** | 看模型错在哪些样本、哪些标签,而不仅是一个 F1 | worker 评估时落 per-sample(已有 uri 字段);新增 `GET /eval-runs/{id}/samples` 分页;UI 表格 + 按"预测≠真值"过滤 | M | **P0** |
| 2.2 | **混淆矩阵 / PR 曲线 / 阈值调优** | 分类、句对场景定位偏差,选最佳阈值 | 评估器多产出概率;UI 画混淆矩阵热力图、PR 曲线、阈值滑块联动指标 | M | P1 |
| 2.3 | **模型版本对比(diff)** | 并排比 V1 vs V2 的指标 + 同一批样本的预测差异 | 复用同测试集的多次 `EvalRun`,UI 做并排/差异视图 | M | P1 |
| 2.4 | **上线回归门禁** | 提升到"生产"前自动跑评估,低于基线则拦截 | 阶段提升时触发评估并比对 prod 基线;不达标 `code` 拦截 + 提示 | M | P1 |
| 2.5 | **分片评估(按 slice)** | 按业务维度(渠道/长度/类别)分组看指标,发现长尾问题 | 评估支持按列分组聚合 | S | P2 |

## 推荐落地顺序

**P0(先做,补齐"能对外用 + 能排障 + 能看懂"的硬缺口):**
1. API Key + 推理调用日志/用量(1.1 / 1.2)——让模型真正可被外部业务调用并可观测
2. 逐样本评估明细(2.1)——字段已就绪,最快见效
3. 审计日志 + 事件通知(5.1 / 5.2)——合规与"训练完自动通知"
4. 一键回滚部署(6.3)、训练队列(4.2)——线上稳健性
5. 数据集画像(3.1)——上传即体检

**P1(增强深度与自动化):** 限流/批量推理、混淆矩阵/阈值调优、版本对比、回归门禁、版本 diff、标注工作流、超参搜索、定时重训、审批流、血缘、SDK。

**P2(规模化与生态):** 金丝雀/多副本、漂移检测、主动学习闭环、断点续训、项目多租户、Prometheus、HF Hub 拉模型等。

---

## 一句话总结

平台的"训练 → 评估 → 部署"主干已经完整。下一步价值最大的不是再加任务类型,而是把它从**"能训练出模型"**推进到**"能把模型当作可被外部安全调用、可观测、可治理、可持续迭代的生产服务"**——即 P0 的 API Key + 调用观测 + 评估明细 + 审计通知 + 回滚。
